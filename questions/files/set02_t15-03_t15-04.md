# Question Set 02 — t15-03, t15-04

---

## Questions

### t15-03
A developer fine-tunes Ruby on 200 episodes from the episode store. After fine-tuning, Ruby's verification catch rate drops from 30% to 5%. Is this good news or bad news, and what's the one thing you'd check before celebrating?

### t15-04
Explain why running the benchmark with `--auto` but without `--readonly` gives misleading results for measuring Ruby's reasoning ability, even if you only look at text answers.

---

## t15-03 — Answer

**It could be good news — or it could be catastrophic overfitting. The drop alone is ambiguous, and that ambiguity is exactly why you must not celebrate yet.**

### Why it could be good news

The verification gate's catch rate measures how often `verifyRubyAnswer` flags Ruby's answers as INVALID and escalates to the large model. A 30% catch rate means the gate was rejecting nearly one in three of Ruby's answers — usually for fabrication or contradiction with tool history. If fine-tuning genuinely taught Ruby to answer correctly and consistently, fewer answers would contradict the tool history, the gate would flag fewer of them, and the catch rate would naturally fall. In that happy case, 5% means "Ruby is now right 95% of the time and the gate rarely has to intervene" — the entire point of the small-model routing system.

### Why it could be catastrophic overfitting

The 200 episodes were drawn **from the same episode store that feeds the benchmark**. If the fine-tuning set overlaps with the evaluation set — i.e., Ruby was trained on answers to the same questions it is now being scored on — then it has memorized the expected answers rather than learned to reason. In that scenario the catch rate drops because the fine-tuned model regurgitates cached correct answers (which happen to pass verification), but it has zero generalization ability. It will collapse on any question it wasn't trained on.

Worse, the fine-tune could have also taught Ruby to **mimic the surface pattern of a verified-correct answer** — the right keywords, the right confidence tone, the right structure — which fools the gate's `complete()` cross-check even when the underlying reasoning is absent. The gate only compares the answer against tool history; it cannot detect a confident-sounding memorized answer that happens not to contradict the tool calls.

This is the classic train/test contamination failure: the metric you're celebrating (low catch rate) dropped precisely *because* the model saw the test set during training, not because it got smarter.

### The one thing to check before celebrating

**Re-evaluate the fine-tuned Ruby on a held-out set of questions that were explicitly excluded from the fine-tuning corpus.** You need a clean separation between the training episodes and the evaluation episodes:

1. **Hold out a partition** — split the episode store (or use a fresh batch of questions) so the 200 fine-tuning episodes and the benchmark evaluation set are **disjoint**. No episode used in fine-tuning should appear in evaluation, and ideally the question families shouldn't overlap either.
2. **Measure catch rate on the held-out set only.** If it stays at ~5% on questions Ruby has never seen, that's genuine improvement — the fine-tune taught transferable reasoning and you can celebrate. If it jumps back up to ~30% (or higher) on held-out questions, the low catch rate was pure memorization — the model overfit to its training data and the fine-tune produced an illusion of competence that will collapse in production.
3. **Sanity-check the absolute answer quality**, not just the gate's pass/fail rate. A model that always outputs the empty string or a generic platitude passes the verification gate trivially (nothing to contradict in the tool history) but catches nothing and answers nothing. The catch rate measures gate interventions, not answer correctness — a near-zero catch rate is only meaningful if the unflagged answers are actually right.

**Verdict:** The drop from 30% to 5% is **ambiguous** — it is good news only if it generalizes to held-out questions the fine-tuned model never saw. The one mandatory check before celebrating is **evaluating the catch rate on a strict train/test partition** to rule out overfitting to the fine-tuning corpus. If the fine-tuning episodes leaked into the benchmark evaluation set, the 5% is meaningless memorization, not real improvement.

## t15-04 — Answer

**Because without `--readonly`, Ruby is allowed to execute tool calls that change its information state — and those tool calls silently feed information into the final text answer, inflating the reasoning score by conflating reasoning with retrieval.**

### What `--auto` does

`--auto` removes the human confirmation step for tool calls. The agent loops autonomously: it calls tools, reads results, and decides its next action without waiting for user approval. This is necessary for headless benchmark runs (there's no human at the terminal to approve `read_file` or `search_code`). But `--auto` only governs *whether* tools execute without confirmation — it says nothing about *which* tools are available or *what information* flows into the answer.

### What `--readonly` restricts

`--readonly` confines the agent to read-only tools — `read_file`, `list_dir`, `search_code`, `git_status`, `git_diff`. No file writes, no shell execution, no mutations. Without it, `--auto` leaves the agent with the **full toolset**, including the ability to read every file in the project. That sounds harmless for a text-answer benchmark, but it is not.

### Why the text answer is contaminated

The scorer (`score_keyword` for Tier 15) checks whether key concepts appear in the final text answer. It does not — and cannot — inspect *how that knowledge got there*. Consider a question like "Explain why the competence gate can cause a starvation loop." There are two ways Ruby's answer could contain the phrase "frozen at its last recorded rate":

1. **Pure reasoning** — Ruby synthesizes the answer from its own parametric knowledge of the Ruby Alternator architecture, producing a correct causal explanation with no external help.
2. **Retrieval-assisted** — Ruby calls `read_file` on `src/ruby/alternator.ts` (or `search_code` for "competence"), reads the actual implementation and comments, then paraphrases the source code into the answer.

Both produce a text answer that passes the scorer's keyword check with the same verdict: "correct." But they measure fundamentally different capabilities. The second path is not reasoning at all — it's the agent finding the answer in the codebase and copying it. Without `--readonly`, the benchmark cannot distinguish case 1 from case 2.

This is why the absence of `--readonly` makes results **misleading, not just noisy**:

- **It systematically inflates scores.** A model that is weak at reasoning but good at navigating the filesystem will score well — it just reads the answer out of the source tree. The benchmark would report high "reasoning ability" that is actually lookup ability.
- **It makes results non-comparable across runs and models.** If the codebase changes between sessions (files added, renamed, refactored), a model's retrieval-assisted score changes for reasons that have nothing to do with its reasoning. Session-to-session comparison becomes meaningless because the "open book" the model is reading from keeps changing.
- **It defeats the experiment's purpose.** The benchmark exists to measure whether *engineering reasoning accumulates* over sessions — i.e., whether the model's internal capability improves. But `--auto` without `--readonly` measures whether the model can find the answer in the *current* filesystem, which is a function of the codebase, not the model. A controlled experiment requires controlled information access: to measure reasoning, you must hold the information available to the model constant across all runs.

### The core principle

**Reasoning is what the model can produce from its own knowledge. Retrieval is what it can copy from the environment. A benchmark that conflates the two measures neither.** `--auto` without `--readonly` lets tool calls — specifically file reads — inject information into the answer that the model could not have produced alone. The scorer, seeing only the final text, credits that information as if it were reasoning. The result overstates Ruby's cognitive ability and understates the gap between "knows the answer" and "can find the answer."

**Verdict:** Running `--auto` without `--readonly` gives misleading results because the agent's tool calls (especially file reads) change what information goes into the final answer. The text-answer scorer cannot see the provenance of each concept — whether it came from the model's parametric reasoning or from a `read_file` call — so it conflates genuine reasoning with retrieval. A controlled measurement of reasoning ability requires controlled information access, and `--readonly` is the minimum constraint that keeps the benchmark measuring the model rather than the filesystem.
