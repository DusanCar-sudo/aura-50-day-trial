# Question Set 01 — t15-01, t15-02

---

## Questions

### t15-01
Aura's `frequency_penalty` and `presence_penalty` are both set to 0.3. A user complains that Aura keeps repeating the same file path in a long refactoring session. Is this a penalty tuning problem, a context problem, or something else entirely? Diagnose precisely.

### t15-02
The episode store records `rubyAttempted=true` when Ruby tries a task. But if Ruby is skipped due to competence gating, `rubyAttempted=false`. Over 50 days, what happens to the competence score for a task category Ruby is never allowed to attempt?

---

## Answers

### Answer t15-01

**It is neither a penalty-tuning problem nor primarily a context problem — it is a tool-call repetition problem with a different root cause, and tuning penalties will not fix it.**

Here is the precise diagnosis:

1. **Penalty parameters only apply to generated output tokens, not to tool-call arguments.** `frequency_penalty` (0.3) and `presence_penalty` (0.3) operate inside the sampling layer of the completion — they upweight/downweight logits for tokens already produced in the *text* of the current generation. A repeated file path inside a `read_file` or `edit_file` tool-call payload is an argument the model emits as a structured function call. The penalty signal does reach the tokens of that path, but only weakly: in a long session the path appears dozens of times in the *context* (prior turns), and the penalty is computed over the current generation window, not the full conversation history. More importantly, the repetition is not a sampling artifact — it is the model *re-deciding* to operate on the same file because its context tells it that file is still relevant.

2. **The real root cause is context saturation / loss of forward progress tracking.** In a long refactoring session the context window fills with the same file's contents across many turns. As the window approaches its limit, the model loses track of what it has *already done* (because completed edits get compacted or scrolled out) while the salient file path stays visible. The result: the model re-reads or re-targets the already-handled file because, from its vantage point, that file still looks like the active unit of work. This is a *context* problem in the broad sense, but not the kind "more penalties" or "more context" alone solves — it is a **working-set / state-tracking** problem.

3. **Why increasing the penalty is the wrong lever.** Cranking `frequency_penalty` higher to suppress the repeated path will degrade legitimate behavior: in a refactor of one file, the path *should* appear repeatedly across tool calls. Penalizing it would push the model toward inventing *different, wrong* paths just to avoid the penalty — a worse failure mode.

**The correct interventions:**
- **Summarize completed work** so the model sees "file X already done" rather than re-encountering it as open work (i.e., compact with an explicit *state delta*, not just a content recap).
- **Cap re-reads** of the same file within a session (the harness can detect "read_file(X) called N times" and inject a directive).
- **Watch the context window**: when the conversation approaches the window limit, `/compact` *before* the model starts looping, and ensure the compaction preserves the task checklist, not just code snippets.

**Verdict:** Something else entirely — a tool-call/working-set state-loss problem, not a sampling-penalty problem. Penalty tuning is a misdiagnosis.

### Answer t15-02

**The competence score for that category freezes at its last recorded value and never updates — creating a self-reinforcing feedback loop that permanently locks Ruby out of the category.**

Precise diagnosis:

1. **The score never updates.** Competence is computed as `successRate = successes / attempts` over recent episodes in that category (gated by `minAttempts`, default 3). When `rubyAttempted=false`, no attempt is recorded — neither a success nor a failure. Over 50 days the numerator and denominator both stay unchanged, so `successRate` remains frozen at whatever value caused the gate to close in the first place. Zero new evidence enters the store.

2. **The gate is self-reinforcing (a starvation loop).** The sequence is:
   - Category X drops below `competenceThreshold` (0.7) after `minAttempts` failures.
   - Gate closes → `rubyAttempted=false` for every future X task.
   - No new attempts → score never moves → gate never reopens.
   
   This is a *deadlock by design*: the competence gate both *consumes* evidence (to make its decision) and *prevents* the creation of new evidence (to reverse that decision). The 50-day wait produces exactly zero learning.

3. **Worst-case pathological outcome.** If the original score that closed the gate was a small-sample fluke (e.g., 3 early failures from a transient issue — a bad model checkpoint, a prompt regression, cold start), that fluke is now *permanently enshrined*. The category is condemned for the entire trial on the basis of `minAttempts` (3) data points, with no mechanism for redemption. Conversely, if Ruby *would* have improved (fine-tuning, prompt fixes, better tooling), that improvement is invisible and unmeasurable because the gate starves it of any opportunity to show it.

4. **The score also doesn't *decay* in the absence of data.** Because the rate is a static ratio over recorded attempts, 50 days of inactivity looks identical to 50 days of competent work — the number simply doesn't move. There is no "staleness" or time-weighting to signal that the score is stale and should be re-probed.

**What to do about it:**
- **Exploration / re-probing:** periodically force `rubyAttempted=true` on a held-out fraction of gated categories (an epsilon-greedy probe) so the score receives fresh evidence and can recover. Even one forced attempt every N tasks breaks the deadlock.
- **Decay toward the mean (or toward neutral):** if a category has had no attempts for D days, shrink the weight of its old sample toward the prior, so a small early failure doesn't dominate forever.
- **Separate "gate decision" from "evidence recording":** even when the gate routes to the fallback model, occasionally *also* let Ruby attempt the task in shadow (without affecting the user) and record the outcome, so the competence signal stays alive.
- **Reserve capacity for recovery:** treat `minAttempts` as a floor for *opening* a gate, not just for *closing* it — require a fresh sample before re-admission, but guarantee that fresh sample actually happens.

**Verdict:** The competence score for the gated category is *frozen at its last recorded rate and never updates*, the competence gate becomes self-reinforcing, and Ruby never gets a chance to recover or demonstrate improvement. This is a classic starvation loop: the gating mechanism, by preventing attempts, also prevents the only thing that could change the gate's verdict.
